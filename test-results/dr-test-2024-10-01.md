RPO: WAL replay got data to within 8 minutes of shutdown. RPO target was 15 minutes. **PASS** — but this was almost invalidated by the backup issue. If the backup had been unrecoverable, we'd have been looking at a 24-hour RPO from the previous night's full backup.

## root causes

**Primary:** The PostgreSQL backup cron job was writing to `/var/lib/postgresql/backups/` on the local VM disk instead of the NAS mount at `/mnt/nas-backup/`. The NAS mount entry existed in `/etc/fstab` but the mount had failed silently after a kernel update two weeks earlier. The backup script didn't check if the destination was mounted — it just wrote to whatever path existed, which happened to be a local directory created as a mount point.

**Secondary:** The DR runbook didn't have a step to verify the backup file exists on the NAS before starting the restore procedure. We went straight to restore and only discovered the problem when the file wasn't there.

**Contributing factor:** WAL archiving was also pointing to the same NAS path and had the same issue, so there were no WAL files on the NAS either. Everything was on local disk.

## what went well

Network and DNS failover were actually faster than expected. The pfSense config backup/restore process worked perfectly — pfSense had auto-backed up its config on the previous day's config change, and restoring it on site-b took under 5 minutes.

Web server failover was straightforward because the VM snapshot on site-b was current. The database connection string update was the only manual step and it took 2 minutes.
